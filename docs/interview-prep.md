# 项目面试准备

## 1 分钟项目介绍

这是一个 AI 算法刷题教练系统，主要面向 LeetCode Hot100 刷题和面试准备。

系统内置 100 道 Hot100 题目，支持用户登录、刷题进度记录、错题本和标签掌握度统计。推荐模块不是简单按顺序推题，而是先根据用户错题和标签掌握率做规则召回，再结合用户画像生成推荐理由、训练重点和学习计划。

AI 部分主要做两件事：第一，AI 聊天会注入当前题目上下文和用户学习画像，回答会更贴合用户薄弱点；第二，用户提交错误代码或错误描述后，后端会结合题目模式、核心思路和常见坑点调用大模型做结构化错因分析，分析结果会保存到错题本里，后续推荐和画像都可以复用。

工程上，我没有把模型调用直接写在 Controller 里，而是拆出了 AI 错因分析服务，包含 prompt 构建、JSON 解析校验、失败修复、兜底降级、调用日志记录和结果落库。

## 30 秒精简版

我做的是一个 AI Hot100 刷题教练系统。它会记录用户刷题进度，统计标签掌握度和错题数据，然后用规则召回候选题，并生成推荐理由和训练重点。用户做错题后，可以提交错误代码或错误描述，后端会结合题目上下文调用大模型生成结构化错因分析，并把错因沉淀到错题本和学习画像里。后续 AI 聊天和推荐都会使用这份画像。

## 架构讲法

```text
前端 Vue
  -> REST / SSE
后端 Spring Boot
  -> AuthService 做登录认证
  -> Hot100ProblemLoader 加载题库
  -> Hot100ProgressService 做进度、错题、掌握度、推荐
  -> Hot100WrongAnalysisService 做 AI 错因分析治理
  -> AiCodeHelperService 封装大模型调用
数据库 MySQL
  -> 用户、会话、进度、错因、AI 调用日志
Redis
  -> 缓存题库、推荐、掌握度等热点数据
```

## 推荐系统怎么讲

推荐不是直接让大模型从题库里选题，而是混合推荐：

1. `tagMastery` 统计每个标签的总题数、练习数、掌握数、错误数和掌握率。
2. `weakTags` 找出用户薄弱标签。
3. `recommendNext` 根据薄弱标签召回未完成题，如果不够再补充未掌握题。
4. `aiRecommendations` 结合薄弱标签、掌握标签、近期错题生成推荐理由、训练重点和教练总结。

可以这样说：

> 我把推荐拆成规则召回和 AI 解释两层。规则召回保证结果稳定、可控，不会推荐不存在或已掌握的题；AI 解释层负责把用户画像转成可读的推荐理由和训练重点。

## AI 错因分析怎么讲

流程：

```text
用户提交错误代码 / 错误描述 / 备注
  -> 后端根据 problemSlug 查询题目上下文
  -> 构造包含题目模式、核心思路、常见坑点的 prompt
  -> 调用大模型生成 JSON
  -> 后端解析校验 JSON
  -> 失败时调用修复 prompt 重试一次
  -> 仍失败时返回保守兜底结果
  -> 保存 wrongReason / knowledgePoint / aiFeedback / nextAction
  -> 写 ai_call_log
```

## 面试官问：这不就是调用 API 吗？

回答：

> 大模型确实是通过 API 调用的，但项目重点不是训练一个模型，而是把大模型能力工程化接入刷题业务。我没有直接在 Controller 里调用模型，而是做了一个 AI 调用治理层。后端会先构造题目上下文和用户错误证据，再要求模型输出固定 JSON；之后会做 JSON 解析校验、失败修复、兜底降级、调用日志记录和错因落库。AI 分析结果不是一次性展示，而是会进入错题本和用户画像，后续推荐和聊天都会复用。

补一句：

> 所以大模型只是推理引擎，后端负责上下文组织、稳定性、可观测性、数据持久化和业务闭环。

## 面试官问：为什么不让前端直接调模型？

回答：

> 首先 API Key 不能暴露在前端。其次 prompt 构建依赖题目上下文、用户画像和服务端数据，应该由后端统一组织。第三，模型输出需要校验、修复和落库，如果放在前端只能得到一次性文本，没法沉淀成用户学习数据。

## 面试官问：推荐是真的 AI 吗？

回答：

> 推荐主体是混合推荐，不是完全让大模型决定。规则层负责候选题召回，保证稳定性和可解释性；用户画像层统计标签掌握度、薄弱标签和近期错题；AI 层负责把这些数据转成教练总结、推荐理由和训练重点。这样比完全依赖模型推荐更可控，也更适合工程落地。

## 面试官问：AI 输出不稳定怎么办？

回答：

> 我对 AI 输出做了治理。错因分析要求模型返回固定 JSON，后端会解析和校验四个必填字段。如果解析失败，会调用一个 JSON 修复 prompt 再试一次；如果仍然失败或模型不可用，就返回保守兜底结果，保证用户流程不中断。同时每次调用都会写 `ai_call_log`，记录耗时、是否成功、是否修复、是否兜底和错误信息。

## 面试官问：你项目里 Redis / 异步任务有什么用？

回答：

> Redis 用来缓存 Hot100 题目列表、题目详情、推荐结果、学习计划和标签掌握度，降低重复计算和数据库压力。异步任务用于推荐和学习计划这种可能较慢的计算场景，前端可以提交任务后轮询结果，避免长时间阻塞请求。

## 面试官问：项目最大改进空间是什么？

回答：

> 现在错因分析已经能结构化落库，但错因类型还可以继续枚举化，比如边界错误、去重错误、状态转移错误、复杂度过高等。后续我会把错因类型和推荐系统结合起来，从按标签推荐升级为按“标签 + 高频错因”推荐专项训练题。

## 演示顺序

1. 登录账号。
2. 打开 Hot100 题库，展示 100 道题和筛选。
3. 选择一道题，保存进度为“做错”。
4. 粘贴一段错误代码或错误描述，点击 AI 分析错因。
5. 展示错因、知识点、AI 建议、下一步行动。
6. 打开错题本，展示错因沉淀。
7. 返回列表，展示薄弱标签、AI 推荐和训练重点。
8. 打开聊天，选择当前题目，说明 AI 聊天会注入题目上下文和用户画像。
